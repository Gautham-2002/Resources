Section 1 — Foundations

What is an LLM?

A stateless function. Text in → text out. No memory, no actions, no side effects.

Every Call is Independent

tokens in
→
probability distribution
→
tokens out

The model doesn't "know" it's in a conversation.
You reconstruct that illusion by prepending history to every prompt.

🧠 Key Insight

There is no continuity between calls. Everything that looks like memory, reasoning, or agency is your code constructing an illusion and feeding it back to a stateless function.

Once you internalize that, the entire design space of agents becomes much clearer — and the failure modes become obvious.

What is Tool Calling?

Adding tools gives the LLM the ability to act — but it's still not an agent. It's a tool-augmented LLM call.

💡 Critical distinction: The LLM suggests a tool call. Your runtime code executes it. The LLM never touches external systems directly — it only produces structured text that your code interprets.
Section 2 — Agents

What is an Agent?

An agent = LLM + tools + a control loop where the LLM decides what to do next.

The Agent Loop

User gives task
↓
LLM decides next action
↓
Call a tool?
Final answer?
↓ tool result
Append result to history
loop back → LLM decides again

🔑 The Key Word is "Decides"

The LLM chooses which tool to call, inspects the result, and decides whether to call another tool or stop. That loop runs until the LLM says "I'm done."

📋 Example

"What is Anthropic's latest model and cost?"

  • LLM decides → search "Anthropic latest model"
  • Gets result → decides → search "pricing"
  • Gets result → decides → final answer

The LLM decided to search twice. That wasn't in your code.

The LLM doesn't know it's in a loop. It only knows because you wrote the history into the context.
Section 3 — Agent Failures

Why Agents Fail

Five critical failure modes you must understand before deploying agents.

🎯 1. Unbounded Action Space

Give an agent 50 tools and it will explore paths you never anticipated. More tools = more ways to go wrong. The agent might call delete_user when you wanted search_docs.

💥 2. Irreversible Actions

An agent that sends emails, writes to production databases, or triggers deployments can cause real, unrecoverable damage. The LLM doesn't understand consequences.

♾️ 3. No Loop Termination

Agents get stuck in loops — repeating the same tool calls, hallucinating that they've completed the task, or endlessly "refining" their work. Without a hard stop, it runs forever.

📦 4. Context Window Overflow

As the agent loops, each tool result gets appended to the history. Context fills up, earlier reasoning gets pushed out, and the agent starts making contradictory decisions.

🚨 5. No max_iterations Hard Limit

In production, this is non-negotiable. Without a hard iteration cap, a stuck agent burns tokens, money, and compute indefinitely. Every production agent must have a kill switch.

Section 3 — Agent Failures

How to Defend Against Each Failure

🎯 Unbounded Action Space → Bounded Toolkit

Only expose the minimum tools the agent needs for the specific task. A research agent gets search and summarize — not send_email or delete_file.

Fewer tools = fewer failure paths = more predictable agent.

💥 Irreversible Actions → Approval Gates

Mark destructive tools (email, DB writes, deployments) as requiring human approval before execution. Use dry-run mode during development.

The agent proposes, the human approves.

♾️ No Termination → Hard Iteration Limit

Set max_iterations (e.g., 10–15). After that many loops, force-stop the agent regardless of state. Also add a wall-clock timeout (e.g., 120 seconds).

Always have two limits: iteration count AND time.

📦 Context Overflow → Auto-Compression

Track token usage. When context reaches ~80% of the limit, automatically summarize older messages to free space while preserving key facts and decisions.

Monitor → warn at 80% → compress → continue.
⚡ Production rule: Every agent in production must have: bounded tools, approval gates for destructive actions, max_iterations, timeout, and context tracking. No exceptions.
Section 4 — Workflows

What is a Workflow?

You (the developer) decide the sequence. The LLM fills in content, not structure. No autonomous decision-making.

📋 Workflow

Control flow lives in your code. Steps are hardcoded. The LLM is just a function being called at each step.

Step 1: Summarize doc
↓ always
Step 2: Extract entities
↓ always
Step 3: Write report
  • Deterministic & debuggable
  • Fewer LLM calls = cheaper
  • Predictable failure modes
  • Full observability and tracing
VS

🤖 Agent

Control flow is decided by the LLM at runtime. Code provides tools and enforces the loop; the LLM drives it.

LLM decides what to do
↓ maybe tool A?
Execute → inspect result
↓ maybe tool B?
LLM decides: done
  • Flexible & adaptive
  • More LLM calls = expensive
  • Opaque reasoning → hard to debug
  • Can waste tokens exploring dead ends

Section 5 — Decision Framework

When to Use What

Can the task be solved with a single prompt?
✅ YES → Plain LLM call. Stop here.
❌ NO → Continue ↓
Are the steps known and fixed before runtime?
✅ YES → Workflow. Predictable, auditable, cheap.
❌ NO → Continue ↓
Do you not know how many steps or which tools are needed?
✅ YES → Single Agent. But add guardrails first.
❌ Still too complex → Continue ↓
Is the task too large for one context window, or are sub-tasks parallelisable?
✅ YES → Multi-Agent. But only if single agent genuinely can't handle it.
⚠️ The #1 mistake: Jumping straight to multi-agent because it sounds more powerful. A well-designed single agent handles most real-world tasks. Add agents only when you hit a concrete limitation.
Section 6 — Multi-Agent

What is Multi-Agent?

Multiple independent agent loops, each with their own LLM, memory, and tools, where one can direct or spawn another.

🎯 When to Use Multi-Agent

  • Task is too large for one context window
  • Sub-tasks are genuinely parallelisable
  • You need specialisation (coder, tester, reviewer)

⚠️ The Real Cost

  • More latency (serial agent calls)
  • More cost (multiple LLM calls per agent)
  • Harder to debug (opaque reasoning × N agents)
  • Error propagation across the chain
Section 6 — Multi-Agent

Multi-Agent Challenges

When you put multiple agents in a pipeline, these problems emerge.

📦 1. Context Explosion

Each agent appends its output to the prompt for the next agent. By agent 3, the prompt is enormous. By agent N, you've hit the context limit.

Agent 1
2K tokens
→
Agent 2
5K tokens
→
Agent 3
9K tokens
→
Agent N
💥 Limit!

🔄 2. Error Propagation

Agent 1 hallucinates → Agent 2 builds on the hallucination → Agent 3 makes it worse. Errors compound across the chain. There's no self-correction between agents.

⏱️ 3. Latency Stacking

3 agents × 5 tool calls each × 2s per call = 30 seconds minimum. Sequential agents multiply latency linearly. Users won't wait.

💸 4. Cost Amplification

Each agent loop compounds token usage. Input tokens grow with each agent. A 3-agent pipeline with 5 turns each = 15+ LLM calls, each with growing context.

🔍 5. Debugging Opacity

When 3 agents each make 5 decisions, you have 15 opaque reasoning steps. Finding which agent made the wrong decision at which point is like finding a needle in a haystack. Traditional debugging doesn't work.

Section 6 — Multi-Agent

How to Solve Each Challenge

📦 Context Explosion → Summarize Between Agents

Before passing one agent's output to the next, compress it. Summarize to keep only key findings, decisions, and data points. Discard verbose explanations.

Agent 1
3K out
→
🗜️ Compress
→ 500 tokens
→
Agent 2
gets 500

Also consider: selective context (only pass what the next agent needs) and shared memory (KV store instead of prompt stuffing) and token budgets (enforce max per step).

🔄 Error Propagation → Validation Gates

Add an evaluation step between each agent. Before Agent 2 receives Agent 1's output, a validator checks: Is this factually consistent? Does it answer the question? Are there hallucinations?

Agent 1
→
✓ Validate
→
Agent 2

The validator can be an LLM call, a rule-based check, or both.

⏱️ Latency Stacking → Parallelise + Timeout

Use Fan-out for independent sub-tasks — run them concurrently instead of sequentially. Set per-agent timeouts. Use cheaper/faster models for intermediate steps that don't need full reasoning power.

3 parallel agents in 2s each = 2s total, not 6s.

💸 Cost Amplification → Tiered Models + Caching

Not every agent needs GPT-4. Use smaller models for classification, extraction, and validation. Cache frequent tool results so agents don't re-fetch the same data. Set token budgets per agent.

Orchestrator: GPT-4. Workers: GPT-4o-mini. Saves 90% cost.

🔍 Debugging Opacity → Structured Observability

Assign a trace ID to each multi-agent run. Log every agent's input, tool calls, decisions, and output with structured JSON. Use a correlation ID to link all agents in a single run. Build dashboards to visualise the flow. This turns an opaque chain into a debuggable pipeline.

Section 7 — Orchestration

What is Agent Orchestration?

The mechanism by which agents are coordinated. It's the routing layer.

🎯 The Core Idea

Agent orchestration = something is managing the flow. It applies to both single-agent (orchestrating tool calls) and multi-agent (orchestrating which agent does what).

Section 8 — Production

Production Best Practices

Essential patterns for deploying agents that don't bring down your infrastructure.

🔄 Retry + Exponential Backoff

When LLM calls fail, retry with increasing delays. Prevents overwhelming a struggling service.

Fail → wait 1s
↓
Fail → wait 2s
↓
Fail → wait 4s
↓
Success ✓

🔌 Circuit Breaker

If a service keeps failing, stop calling it entirely. Prevents cascading failures.

↓ cooldown

🪣 Rate Limiting
